## Summary

### Abies Automaton v2.0.0 vs Blazor WASM v10.0.0

Abies Automaton is **faster than Blazor in every duration benchmark** — by margins ranging from 26% to 83%. The geometric mean across all 9 duration benchmarks shows Abies at **1.00× vs Blazor at 1.98×** (Blazor is ~2× slower overall).

Key wins:
- **Select row**: 83% faster (13.8 ms vs 82.5 ms) — view cache + ReferenceEquals bailout
- **Swap rows**: 64% faster (33.8 ms vs 94.6 ms) — LIS algorithm + head/tail skip
- **Clear rows**: 50% faster (18.3 ms vs 36.5 ms) — O(1) clear fast path
- **Create 1k**: 39% faster (51.7 ms vs 84.9 ms) — SetChildrenHtml batch patch

Abies also has a **smaller bundle** (-17% compressed) and **faster first paint** (-26%).

The one area where Blazor has an edge is **run+clear memory** (49.4 MB vs 58.6 MB, +19%), likely due to Blazor's more aggressive GC behavior during clear operations.

### Abies Automaton v2.0.0 vs Picea Abies v1.0.152

The Automaton rewrite delivers **dramatic improvements** over the original Picea implementation:

- **Select row**: 87% faster (13.8 ms vs 102.9 ms) — was the biggest bottleneck in Picea
- **Swap rows**: 63% faster (33.8 ms vs 92.2 ms) — LIS fix + binary batching
- **Remove row**: 68% faster (20.7 ms vs 65.7 ms) — optimized keyed diffing
- **Update 10th row**: 37% faster (67.1 ms vs 106.7 ms) — generic memo key equality
- **First paint**: 41% faster (57.7 ms vs 97.5 ms) — loading placeholder optimization

The geometric mean shows **Picea at 1.85× slower** than Automaton across all duration benchmarks.

### Architecture Insight

These results validate the Automaton kernel's approach:

1. **Pure MVU can beat component-based** — Abies's functional architecture outperforms Blazor's stateful component model, disproving the assumption that MVU is inherently slower.

2. **Binary batching + innerHTML** — The SetChildrenHtml optimization (single `innerHTML` instead of N DOM ops) is the key enabler for the create/replace wins.

3. **View cache + ReferenceEquals** — The lazy memo cache enables O(1) bailout for unchanged subtrees, explaining the massive select/swap improvements.

4. **Head/tail skip + LIS** — The keyed diff algorithm minimizes DOM moves, critical for swap and partial-update scenarios.
